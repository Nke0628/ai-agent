## 1. プロジェクト概要

### 1.1 背景・課題

- LLMを活用したチャットアプリの仕組み（ストリーミング応答、会話コンテキスト管理、API連携）を実践的に理解したい。
- 既存のチャットUI（Claude / ChatGPT）と同等の基本体験を、自分でコントロールできる形で構築したい。
- 将来的にRAGやツール利用などのエージェント機能を追加するための土台（ベースアプリ）が欲しい。

### 1.2 目的

Claude APIを利用した、シンプルで快適に使えるWebチャットアプリを構築する。
「AIを使うこと」自体ではなく、**日常的に使える品質のチャット体験を最小構成で実現し、今後の拡張基盤とすること**を目的とする。

### 1.3 ゴールと成功指標（KPI）

| 指標 | 現状値 | 目標値 | 測定方法 |
|---|---|---|---|
| 基本チャット機能の完成 | 0% | 全Must機能の実装完了 | 機能一覧（3.1）との突合 |
| 応答開始までの体感速度 | — | ストリーミング開始2秒以内 | 手動計測・ログ |
| 自分での常用 | — | 週5回以上利用する状態 | 利用ログ |
| 月間API利用料 | — | 3,000円以内 | Anthropicコンソールで確認 |

### 1.4 スコープ

#### スコープ内（アプリに持たせること）
- テキストによるマルチターン対話（ストリーミング表示）
- 会話（スレッド）の作成・一覧・切替・削除
- 会話履歴の永続化（ローカルDB）
- Markdownレンダリング（コードブロックのシンタックスハイライト含む）
- システムプロンプトの設定（アプリ全体で1つ、設定画面で編集可）
- モデル選択（Claudeの複数モデルから選択）

#### スコープ外（今回はやらないこと）
- RAG・ナレッジ検索、ツール実行（Web検索・コード実行等）
- 画像・ファイルの添付、音声入出力
- マルチユーザー対応・アカウント管理（利用者は自分1人）
- 会話の共有・公開機能
- モバイルアプリ（レスポンシブWebで代替）
- LLM出力の自動実行（生成物はテキスト表示のみ。副作用のある操作は一切させない）

### 1.5 前提条件・制約条件

| 種別 | 内容 |
|---|---|
| 前提条件 | Anthropic APIキーを取得済みであること |
| 前提条件 | 利用者は開発者本人1名（ローカルまたは自分専用環境で稼働） |
| 制約条件 | APIキーをブラウザ（フロントエンド）に露出させない。API呼び出しは必ずサーバー側で行う |
| 制約条件 | 月間API利用料は3,000円以内 |
| 制約条件 | 業務上の機密情報・他者の個人情報を入力しない運用とする |

## 2. ユースケース定義

### 2.1 利用者（アクター）

| アクター | 説明 | 想定人数 | 利用頻度 |
|---|---|---|---|
| 利用者（本人） | 質問・相談・文章作成依頼などの対話を行う | 1名 | 日次 |

### 2.2 ユースケース一覧

| UC-ID | ユースケース名 | 概要 | 優先度※ | 自動化レベル |
|---|---|---|---|---|
| UC-001 | 質問応答 | 自由入力の質問にAIが回答する | Must | 全自動 |
| UC-002 | マルチターン対話 | 文脈を踏まえた続きの対話を行う | Must | 全自動 |
| UC-003 | 会話の管理 | 会話の新規作成・一覧・切替・削除を行う | Must | 手動操作 |
| UC-004 | 応答の中断・再生成 | 生成中の応答を停止する / 最後の応答を再生成する | Should | 手動操作 |
| UC-005 | モデル・設定変更 | 使用モデルやシステムプロンプトを変更する | Should | 手動操作 |
| UC-006 | 応答のコピー | 応答内容をクリップボードにコピーする | Could | 手動操作 |

※優先度（MoSCoW）：Must / Should / Could / Won't

### 2.3 対話フロー

代表フロー（UC-001/002）：

```
ユーザー入力
  → 入力検証（空文字・サイズ上限チェック）
  → 会話履歴＋システムプロンプトを組み立ててClaude APIへ送信
  → ストリーミング受信・逐次表示
  → 完了後、会話履歴をDBへ保存
  → （エラー時）エラーメッセージ表示＋再試行ボタン提示
```

### 2.4 エスカレーション条件（人間へ引き継ぐ条件）

利用者＝運用者本人のため、エスカレーションは「アプリからユーザーへの明示」に読み替える。

| 条件 | 挙動 |
|---|---|
| API障害・レート制限 | エラー内容を表示し、再試行を案内する（自動リトライは1回まで） |
| コンテキスト長超過 | 「会話が長すぎる」旨を表示し、新規会話の開始を案内する |
| API利用料の予算超過が近い | Anthropicコンソールの予算アラートで本人へ通知（アプリ外で設定） |

### 3.4 会話・コンテキスト要件

| 項目 | 要件 |
|---|---|
| マルチターン対応 | 同一会話内の全メッセージを文脈としてAPIへ送信する |
| コンテキスト上限対策 | 送信トークンが上限に近づいた場合、古いメッセージから省略する（会話の直近N往復＋システムプロンプトを優先） |
| 会話履歴の保存 | 全会話をローカルDB（SQLite）に無期限保存。ユーザー操作で個別削除可能 |
| 長期記憶 | 会話をまたぐ記憶機能は持たない（スコープ外） |

### 3.5 出力要件

| 項目 | 要件 |
|---|---|
| 回答形式 | 入力言語に追従（日本語入力には日本語で応答）。Markdown形式 |
| 表示方式 | ストリーミングで逐次表示する |
| 不明時の応答 | システムプロンプトで「不確かな場合は推測と明示する / 知らないことは知らないと言う」を指示する |
| AI利用の明示 | アプリ画面に「応答はAIによる生成であり誤りを含む場合がある」旨を常時表示する |

## 4. AIモデル・アーキテクチャ要件

### 4.1 モデル選定

| 項目 | 内容 |
|---|---|
| 利用モデル | 既定：Claude Sonnet（`claude-sonnet-5`）。軽量・低コスト用にClaude Haiku（`claude-haiku-4-5-20251001`）を選択可。タイトル生成はHaiku固定 |
| 選定基準 | 日本語品質、コスト、応答速度のバランス。日常利用の既定はSonnet |
| モデル更新方針 | 新モデル公開時は評価セット（5.2）で確認してから既定モデルを切り替える |

### 4.2 アーキテクチャ構成

```
ブラウザ（Next.js / React UI）
  → Next.js API Route（サーバー側）
      ├ Anthropic API（Messages API, streaming）
      └ SQLite（会話・メッセージ・設定の永続化）
```

| 構成要素 | 内容 |
|---|---|
| フロントエンド | Next.js（App Router）+ React + TypeScript。Markdown表示に react-markdown 等 |
| バックエンド | Next.js API Route。Anthropic公式SDK（`@anthropic-ai/sdk`）でストリーミング呼び出し |
| データストア | SQLite（Prisma等のORM経由）。テーブル：conversations / messages / settings |
| 認証 | なし（ローカル利用前提）。外部公開する場合はBasic認証等を必須とする（7.3） |

### 4.3 プロンプト管理

| 項目 | 要件 |
|---|---|
| 管理方法 | 既定システムプロンプトはリポジトリ内でGit管理。ユーザー編集値はDBに保存 |
| 変更プロセス | 既定プロンプト変更時は評価セット（5.2）で応答品質を目視確認する |

## 5. 品質・評価要件（精度目標）

> 個人開発のため、大規模な自動評価は行わず「小さな評価セット＋目視確認」で品質を定義する。

### 5.1 精度目標

| 評価項目 | 目標値 | 測定方法 |
|---|---|---|
| 基本応答品質 | 評価セット20問で「問題なし」9割以上 | 本人による目視評価 |
| 表示崩れ | Markdown・コードブロックの表示崩れ0件 | 評価セット内の表示確認用プロンプトで確認 |
| ストリーミング安定性 | 途中切断時にエラー表示・再試行が機能する | 手動テスト |
| 履歴の整合性 | 保存・再読込で会話内容が完全一致 | 手動テスト |

### 5.2 評価セット（ゴールデンデータセット）

| 項目 | 内容 |
|---|---|
| 件数・内訳 | 計20件：一般質問10件＋コード生成・表・数式など表示確認5件＋長文/長会話3件＋不適切入力2件 |
| 作成者 | 本人。`docs/eval/` にMarkdownで管理 |
| 更新方針 | 利用中に見つけた失敗事例を随時追加する |

### 5.3 評価の運用

| 項目 | 内容 |
|---|---|
| リリース前評価 | 初回リリース時と、プロンプト・モデル変更時に評価セットを流して目視確認 |
| 継続評価 | 常用の中で気づいた問題をIssue化して改善する |
| ユーザーフィードバック | 専用機能は設けない（利用者＝開発者本人のため） |

## 6. 安全性・ガードレール要件

### 6.1 入力ガードレール

| 項目 | 要件 |
|---|---|
| プロンプトインジェクション対策 | 外部文書・ツールを扱わないため攻撃面は限定的。システムプロンプトとユーザー入力をAPIのroleで明確に分離する |
| 不適切入力の処理 | モデル既定の安全機構に委ねる（追加フィルタは実装しない） |
| 入力サイズ制限 | 1メッセージ50,000文字まで。超過時は送信前にエラー表示 |

### 6.2 出力ガードレール

| 項目 | 要件 |
|---|---|
| 禁止事項 | 追加の出力フィルタは実装せず、モデルの安全機構に委ねる |
| 機密情報の漏洩防止 | 外部データ接続がないため対象外。会話ログはローカル保存のみ |
| 個人情報の取り扱い | 機密・他者個人情報を入力しない運用ルールとする（1.5） |
| XSS対策 | Markdownレンダリング時にHTMLをサニタイズする（生HTML描画禁止） |

### 6.3 行動範囲の制限

| 項目 | 要件 |
|---|---|
| ツール実行権限 | ツールなし。AIはテキスト生成のみで、いかなる副作用も発生させない |
| 実行回数・ループ制限 | 1送信につきAPI呼び出し1回（＋タイトル生成1回）。自動リトライは1回まで |
| コスト暴走防止 | `max_tokens` を設定（既定8,192）。Anthropicコンソールで月次予算アラートを設定 |

### 6.4 責任と透明性

| 項目 | 要件 |
|---|---|
| 免責・利用規約 | 画面フッターに「AI生成のため誤りを含む場合がある」旨を常時表示 |
| 監査可能性 | 全会話がDBに残る。APIエラーはサーバーログに記録する |

## 7. 非機能要件

### 7.1 性能

| 項目 | 要件 |
|---|---|
| 応答開始時間 | ストリーミング開始まで2秒以内（体感目標） |
| UI操作 | 会話切替・一覧表示は1秒以内 |
| 同時利用 | 1ユーザー（同時セッション考慮不要） |

### 7.2 コスト

| 項目 | 要件 |
|---|---|
| API利用料の上限 | 月額3,000円以内 |
| コスト監視 | Anthropicコンソールで利用量を週次確認、予算アラートを設定 |
| コスト最適化 | 長い会話は古いメッセージを省略して送信。タイトル生成等の軽量タスクはHaikuを使用 |

### 7.3 セキュリティ・データ保護

| 項目 | 要件 |
|---|---|
| APIキー管理 | 環境変数（`.env.local`）で管理し、Gitにコミットしない。フロントエンドへ渡さない |
| 認証・認可 | ローカル利用のため認証なし。インターネット公開する場合は認証必須（別途要件化） |
| データの外部送信 | Anthropic APIへの送信のみ。APIデータは学習に利用されない（Anthropicの商用APIポリシーに準拠） |
| ログの取り扱い | 会話データはローカルSQLiteのみに保存 |

### 7.4 可用性・運用

| 項目 | 要件 |
|---|---|
| 稼働時間 | 本人利用時のみ（常時稼働不要） |
| LLM API障害時 | エラーメッセージと再試行ボタンを表示（フォールバックモデルなし） |
| 監視 | 専用監視なし。APIエラーはサーバーログで確認 |
| バックアップ | SQLiteファイルを任意タイミングで手動バックアップ |
